## 5.6 Security Awareness
 
### Security Awareness Program Components
 
| Component | Description |
|---|---|
| Phishing Campaigns | Simulated phishing emails sent to employees. Track click rates. Immediate feedback and training for those who click. Schedule follow-up for repeat offenders. |
| Anomalous Behavior | Risky: modifying host files, uploading sensitive data. Unexpected: login from another country, unusual data transfers. Unintentional: mistyped domain, lost USB drive. |
| Policy/Handbooks | Document security requirements. Publish online. Reference in employee handbook. Make accessible before a user needs them. |
| Situational Awareness | Train users to recognize software attacks (email links, unusual URLs) and physical attacks (unknown USB drives, unlocked doors). |
| Insider Threats | Multiple approvals for critical processes. Monitor files and systems. Make unauthorized changes technically difficult. |
| Password Management | Train on complexity requirements, password managers, and avoiding password reuse across accounts. |
| Removable Media | Warn against unknown USB drives and cables. Enforce policies on personal devices connecting to company resources. |
| Social Engineering | Ongoing training on pretexting, phishing, vishing, and impersonation recognition. |
| OPSEC | Operational security. View security from the attacker's perspective. Identify sensitive data and keep it private before it can be exploited. |
| Hybrid/Remote Work | VPN requirements, personal device policies, physical security at home workspaces, additional endpoint security controls. |
 
### Phishing Campaign Process
 
1. Design a realistic phishing email mimicking a trusted sender or brand.
2. Send to the employee community without prior notice.
3. Track click rates, credential submissions, and attachment opens.
4. Provide immediate feedback and training to users who clicked.
5. Schedule follow-up training for repeat offenders.
6. Report metrics to stakeholders: click rates over time, improvement trends, training completion.
### Security Awareness Program Structure
 
- Development: build a security awareness team. Define roles for training, monitoring, and policy creation. Set minimum awareness levels by job function. Integrate compliance requirements (PCI DSS, HIPAA, GDPR).
- Execution: create training materials. Deploy via classroom sessions, posters, and regular emails. Track participation with automated reporting.
- Metrics: measure performance through phishing click rates, password manager adoption, MFA usage rates, and training completion percentages. Update content in low-performing areas.
- Reporting: initial incidents are training opportunities. Recurring issues require escalation. Long-term monitoring identifies systemic behavioral problems.

## Quick Reference
 
| Concept | Key Facts |
|---|---|
| HIPAA spelling | HIPAA (Health Insurance Portability and Accountability Act). Not HIPPA. Both appear as answer choices. |
| Policy hierarchy | Policy (mandatory direction), Standard (mandatory requirements), Procedure (step-by-step), Guideline (recommended, not mandatory). |
| AUP | Acceptable Use Policy. Defines permitted use of company assets. Limits legal liability. |
| Data owner | Senior executive accountable for the data asset. |
| Data controller | Manages purpose and means of data processing. |
| Data processor | Processes data on behalf of the controller. Often a third party. |
| Data custodian | Responsible for data accuracy, labeling, compliance, and access controls. |
| Qualitative risk | Descriptive. Traffic light grids. Rare/possible/almost certain. |
| Quantitative risk | Numerical. SLE, ALE, ARO formulas. Monetary values for decisions. |
| SLE formula | AV x EF = SLE |
| ALE formula | SLE x ARO = ALE |
| Risk transfer | Cybersecurity insurance. Contracts assigning vendor liability. |
| Risk accept | Acknowledge and document. Cost of mitigation exceeds risk value. |
| Risk avoid | Stop the activity creating the risk. |
| Risk mitigate | Reduce likelihood or impact through security controls. |
| Exemption vs. exception | Exemption: cannot follow a regulation, needs approval. Exception: internal policy deviation with documented justification. |
| RTO | Maximum acceptable time to restore service after failure. |
| RPO | Maximum acceptable data loss measured in time. |
| MTTR | Mean Time to Repair: average time to restore a failed system. |
| MTBF | Mean Time Between Failures: average time between failures. Higher = more reliable. |
| SLA | Minimum service terms: uptime, response time, penalties. |
| MOU | Non-binding agreement. States common goals and intent. |
| NDA | Confidentiality agreement. Formal. Signatures required. Unilateral or bilateral. |
| Bug bounty | Rewards researchers for responsible vulnerability disclosure. Controlled release after fix is deployed. |
| Right-to-audit clause | Legal right to audit a vendor's security posture included in the contract. |
| PCI DSS | Payment card security standard. Annual risk assessments and penetration testing required. |
| GDPR | EU data protection. Right to be forgotten. Applies to all EU citizen data globally. |
| SOX | Public company accounting reform. Financial reporting integrity. |
| GLBA | Financial institution privacy disclosure requirements. |
| White box (known) | Full knowledge. Most efficient. Also called known environment. |
| Gray box (partial) | Partial knowledge. Simulates insider. Also called partially known. |
| Black box (unknown) | No knowledge. Most realistic. Also called blind test. |
| Passive recon | Open-source only. No target interaction. No detection risk. |
| Active recon | Direct target interaction. Port scans, DNS queries. Risk of detection. |
| Phishing campaign | Simulated phishing to employees. Measure click rates. Immediate training for clicks. |
| OPSEC | View security from attacker's perspective. Identify and protect sensitive data proactively. |
